| Action | Risk | Rollback |
|---|---|---|

| `elementor_update_text`, `elementor_update_image`, `elementor_update_button` | medium | **yes** |

The **only** mutating operations are the 3 widget edits; all route through one shared `edit_widget()` (`:133`).

## 2. Per-operation forensics (the 3 mutating edits)

**1. Object type:** a **POST** (page) — `page_id`/`post_id`/`content_id` (`page_id()` `:178`).
**2. Post/meta fields written:** **only `_elementor_data`** (postmeta) via `save_data()` (`:192–195`). The shared `edit_widget` decodes `_elementor_data`, locates the widget by id (recursive `mutate_widget` `:205`), applies a per-op mutator to that widget's `settings`, re-encodes, and `update_post_meta($id,'_elementor_data',wp_slash(json))`.
**3. Mutates `_elementor_data`?** **YES** — it is the sole write target.
**4. Mutates `_elementor_css`?** Only **deletes** it as a cache-bust (`clear_cache` `:197–202`, `delete_post_meta($id,'_elementor_css')` + `files_manager->clear_cache()`). It is a **regenerable cache**, not a data source; never a rollback target.
**5. Mutates page settings (`_elementor_page_settings`)?** **NO.**
**6. Mutates templates?** **NO** (no template ops exist).
**7. Rollback record captured (`store_rollback` `:242`, called from `edit_widget` `:148`):** `before_state = ['data' => $before_json]` where `$before_json = get_post_meta($id,'_elementor_data',true)` captured **before** the mutate (`:141`) — i.e. the **WHOLE pre-edit `_elementor_data` JSON string**. Stored in option `wpcc_elementor_rollbacks`, **FIFO cap 100** (`:248`), autoloaded, O(n) scan to resolve (`:160`). `rollback_id` is surfaced (`:150`).
**8. Rollback full-object?** **YES — whole-document.** `rollback()` (`:155–174`) does `update_post_meta($id,'_elementor_data',wp_slash($before.data))` + `clear_cache`. The entire `_elementor_data` tree is overwritten.
**9. Rollback drift-aware?** **NO.** Unconditional whole-document restore — a concurrent/later edit to **any** widget on the page is silently clobbered (the F-1 pattern PROGRAM-4C flagged: layered widget edits each snapshot the full tree; rolling back one resurrects the entire prior tree).
**10. Tests:** `test-elementor-step96.sh` — seeds a real `_elementor_data` tree, exercises read ops + update_text/image/button + **one button rollback** + cache-bust + error cases (widget-not-found, missing widget_id). **No drift / no sibling-widget preservation / no out-of-order / no malformed-record rollback tests.**

## 4. Residual gaps P4.10 must address (honestly + safely)
- **`_elementor_data` whole-document restore is not drift-aware** → a layered edit to a different widget is clobbered on rollback (F-1). → whole-document, drift-aware restore (refuse-on-drift, never clobber), per-post storage.
- **FIFO-100 autoloaded option** → a surfaced `rollback_id` can be silently evicted; the (potentially large) JSON is autoloaded on every request. → move new records to `PostMetaRollbackStore` (O(1), no FIFO, not autoloaded, GC with the page).
- **No partial/conflict honesty** → a no-op or clobbering rollback returns `restored:true`. → honest conflict (`reversible:false`) on drift; honest handling of malformed/missing.
- **Out of scope (left as-is / honest):** page settings, templates, post fields (not mutated by the runtime → nothing to roll back); `_elementor_css` (regenerable cache); **widget-level granular rollback** (NOT clearly safe under concurrent edits → keep whole-document atomic); clone/import (no such ops).

## 5. Design implication
The single rollback surface — `_elementor_data` as one **atomic whole-document JSON field** — maps directly onto the ACF P4.9 pattern (atomic whole-field value + drift guard + `PostMetaRollbackStore` + `RollbackDelta`). No JSON decomposition, no widget-level rollback. Proceeding to Phase B (risk classification).
